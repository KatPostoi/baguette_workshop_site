# Промпт для запуска ревью `baguette_workshop_site`

## Цель
- [ ] Проведи полное техническое ревью проекта-референса `baguette_workshop_site`.
- [ ] Найди реальные баги, архитектурные разрывы, недоработки, хрупкие места и риски сопровождения.
- [ ] Подготовь результат так, чтобы на его основе можно было собрать общий план доработки в `specs/3_common_plan.md` и затем разложить его по шагам.

## Зафиксированные условия ревью
- [ ] Ревью должно оценивать только текущую реализацию в коде и реальное текущее поведение системы.
- [ ] Разрешено и желательно поднимать `docker`-контур и делать runtime-проверки, если это помогает подтвердить или опровергнуть выводы.
- [ ] Раздел `Admin` является главным приоритетом, но внутри него нужно полноценно проверить все направления: `orders`, `catalog/data`, `audit`, `users`, `teams`.
- [ ] На этом шаге не исправляй код и не делай рефакторинг ради красоты. Нужен именно аудит текущего состояния.

## Контекст проекта
- Frontend: `React + TypeScript + Vite` в `website/`
- Backend: `NestJS + Prisma` в `backend/app/`
- База данных: `PostgreSQL` в `postgresql/`
- Прокси и единая точка входа: `Nginx` в `nginx/`
- Общий запуск стека: `./start-all.sh dev` и `./start-all.sh test`
- Основные frontend admin-маршруты: `/admin/orders`, `/admin/audit`, `/admin/data`
- Основные backend admin API: `/admin/orders`, `/admin/catalog`, `/admin/materials`, `/admin/styles`, `/admin/services`, `/admin/users`, `/admin/audit`, `/admin/teams`
- Демо admin-пользователь из seed: `admin1@baguette.local` / `password`

## Роль
- Ты выступаешь как `senior full-stack reviewer / solution architect`.
- Тебе нужен практический, проверяемый и доказательный аудит, а не абстрактные рекомендации.
- Приоритет у проблем, которые влияют на работоспособность, согласованность системы, безопасность, поддержку и дальнейшую разработку.

## Что не входит в это ревью
- Не оценивай продуктовую полноту вне текущей реализации в коде.
- Не предлагай новые фичи как обязательные, если проблема не вытекает из уже существующего кода, API, схемы данных или runtime-поведения.
- Не уходи глубоко в вкусовые замечания по стилю кода, если они не создают реальный риск.

## Порядок выполнения
- [ ] Изучи структуру репозитория и карту модулей.
- [ ] Зафиксируй ключевые точки входа: frontend routes, backend modules/controllers, Prisma schema, Docker Compose, Nginx.
- [ ] Проверь статическое состояние проекта: зависимости, команды, сборку, `typecheck`, `lint`, потенциальные конфликты в конфигурации.
- [ ] Подними проект там, где это возможно и оправдано, и проверь runtime-контур: frontend, backend, авторизацию, admin-маршруты, ключевые API.
- [ ] Сопоставь сквозные сценарии `frontend -> API -> DB -> infra`.
- [ ] Отдельно проверь связность `Admin`: что доступно в UI, что доступно только через API, что отсутствует, что ломается, что выглядит незавершённым.
- [ ] Сформируй findings только по тем выводам, которые подтверждаются кодом, конфигурацией, сборкой или runtime-проверками.

## Что именно проверить

### 1. Общая архитектура
- Насколько согласованы `frontend`, `backend`, `Prisma schema`, `Docker Compose` и `Nginx`.
- Где проходят реальные границы ответственности между слоями.
- Есть ли дублирование логики, скрытые зависимости, разрывы контракта или хрупкие интеграции.
- Насколько проект пригоден как референс для дальнейшей доработки без предварительной стабилизации.

### 2. Frontend
- Роутинг, guards, авторизация, хранение сессии, refresh-flow, обработка `401/403`.
- Состояния `loading / empty / error / success`.
- Качество API-слоя, типизации, маппинга ответов и обработки ошибок.
- Согласованность между страницами, компонентами и backend-контрактами.
- Явные баги, dead code, хрупкие места, гонки состояний, некорректные assumptions.
- Удобство и предсказуемость admin-сценариев.

### 3. Backend
- Корректность модульной структуры `NestJS`.
- `Auth`, guards, roles, DTO validation, обработка ошибок и консистентность ответов.
- Разделение публичных и admin endpoint'ов.
- Бизнес-правила по заказам, статусам, командам, аудиту, каталогу, материалам, стилям, услугам и пользователям.
- Риски безопасности, утечки чувствительных данных, небезопасные логи, слабые места авторизации и ролей.
- Подозрительные места, которые могут ломать admin-функционал или создавать скрытые регрессии.

### 4. База данных и Prisma
- Соответствие `schema.prisma` текущему домену и API.
- Связи, `nullable`, `unique`, `indexes`, каскады, ограничения и потенциальные проблемы целостности.
- Насколько текущая модель данных реально поддерживает существующие admin-сценарии.
- Риски для будущих миграций, расширения функционала и поддержки данных.

### 5. Admin как главный приоритет

#### 5.1 Orders
- Список заказов, фильтры, диапазоны дат, фильтрация по пользователю и команде.
- Назначение команд, смена статусов, bulk-операции, детализация заказа, timeline.
- Согласованность разрешённых действий между UI, backend и бизнес-правилами.
- Корректность отображения и обновления состояния после действий администратора.

#### 5.2 Catalog / Data
- CRUD для каталога, материалов, стилей и услуг.
- Полнота полей в UI и их соответствие DTO, Prisma и backend-логике.
- Валидация, обработка ошибок, поведение при пустых и некорректных значениях.
- Реальная пригодность страницы `/admin/data` для администрирования без ручных обходных путей.

#### 5.3 Audit
- Получение, фильтрация, пагинация и читаемость аудита.
- Полнота событий по критичным admin-операциям.
- Полезность структуры данных аудита для расследования действий и инцидентов.
- Расхождения между тем, что должно аудироваться по коду, и тем, что реально видно в UI/API.

#### 5.4 Users
- Поиск, фильтрация и отображение пользователей.
- Связность пользовательских данных с ролями, заказами и административными сценариями.
- Достаточность существующего admin-функционала по пользователям относительно того, что уже реализовано в API.

#### 5.5 Teams
- Наличие и качество поддержки команд на backend и во frontend.
- Возможность реально управлять командами через существующий UI.
- Разрывы между API `/admin/teams`, заказами и административным интерфейсом.

#### 5.6 RBAC и связность Admin
- Одинаково ли frontend и backend трактуют права доступа.
- Есть ли функции backend, для которых нет UI, и функции UI, которые не поддерживаются backend.
- Какие части `Admin` нужно стабилизировать в первую очередь, чтобы остальная доработка не строилась на нестабильной базе.

### 6. Инфраструктура и эксплуатация
- `Docker/dev/test` режимы, bootstrap Prisma/seed, healthcheck'и.
- `Nginx` как единая точка входа и корректность проксирования.
- Насколько легко поднять проект с нуля и воспроизвести ключевые сценарии.
- Где технический долг в инфраструктуре будет мешать дальнейшим доработкам и проверкам.

### 7. Качество и поддерживаемость
- Наличие или отсутствие тестов и реальный риск от их отсутствия.
- Документация, пробелы в описании запуска и поведения системы.
- Приоритетные места для стабилизации до начала дальнейшей разработки.

- [ ] Проверить доступность frontend, backend и admin-маршрутов.

- [ ] Проверить ключевые admin-сценарии минимум по одному реальному пути в каждом блоке: `orders`, `catalog/data`, `audit`, `users`, `teams`.
- [ ] Если какая-то runtime-проверка не выполнена, явно указать почему.

## Формат результата
- Сначала выдай только `findings`. Не начинай с общего обзора.
- Отсортируй findings по severity: `critical`, `high`, `medium`, `low`.
- Для каждого finding укажи:
  - область: `frontend` / `backend` / `database` / `infra` / `admin`
  - краткий заголовок
  - почему это проблема
  - как проявляется или чем грозит
  - доказательства: код, конфигурация, команда, лог или runtime-наблюдение
  - точные ссылки на файлы и строки
  - если возможно, шаги воспроизведения
  - рекомендуемое направление исправления

- После findings добавь разделы:
  1. `Verification Performed` — какие команды, проверки и runtime-сценарии были реально выполнены.
  2. `Assumptions` — какие допущения всё же пришлось сделать.
  3. `Open Questions` — только те вопросы, без которых невозможно уверенно оценить уже существующее поведение.
  4. `Quick Wins` — 3-7 самых полезных первых исправлений.
  5. `Admin Priority Map` — что чинить в `Admin` в первую очередь, что во вторую, что позже.
  6. `Readiness Verdict` — можно ли использовать проект как референс для дальнейшей разработки и при каких условиях.

## Ограничения
- Не расплывайся по стилю кода, если нет реального риска или влияния на поддержку.
- Не хвали проект формально.

- Если нашёл гипотезу, но не смог подтвердить её проверкой, пометь её как hypothesis, а не как факт.
